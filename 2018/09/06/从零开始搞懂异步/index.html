从零开始搞懂异步

发表于 2018-09-06 | 更新于 2019-03-11

从零开始搞懂异步

之前小学期学 NodeJS 的时候就被异步搞得很头疼，过几天的 Python 自学报告就打算写异步编程了。现在我打算从最基础的地方开始弄懂异步和与其相关的一些概念。

阻塞与非阻塞、同步与异步、并发与并行

  • 阻塞：程序的某个线程进行了某个调用后如果这个线程要等着运行结果，则称为线程被阻塞。（虽然 CPU 可以把它换下来，但是这个线程依然是阻塞的。）
  • 非阻塞：程序的某个线程进行了某个调用后不等结果出来继续运行，则称为非阻塞。
  • 同步：进行了某个调用后，这个调用不是立即返回的，可能需要等一段时间，一旦调用返回就得到了希望得到的记过，则称这种情况为同步调用。
  • 异步：进行了某个调用后，这个调用立即返回，但是返回值不是希望得到的内容的全部。过一段时间之后可能通过回调等方式处理了真正要得到的记过，这种情况称为异步调用。
  • 并发：设计上能够有同时处理多件事情的能力，宏观上这些事情是同时在做的，但是实际不见得就是物理上同时处理的。
  • 并行：物理上同时做多件事情。

按我的理解，阻塞/非阻塞描述的是程序的状态，同步/异步描述的是接口的行为，并发是一种能力，并行是一种现象。

只有同步阻塞、同步非阻塞（某种基于轮询的搞法）、异步这三种组合。

另外我见到一个说法是同步是不同程序单元进行通信来协调一致的状态，异步是不同程序单元可以不通信就能完成任务的状态。为什么我们需要异步，因为异步可以实现非阻塞。

深究这些定义可能没什么卵用。

异步编程的演化

造成阻塞的原因大部分是等待 IO，如果 IO 是异步的那么阻塞会少很多。

深入理解 Python 异步编程 (上) 这篇博客从头解释了各种异步的实现，最后引出了 Python 中的 asyncio。这一节是对这篇博客的整理，博客中是以下载十篇网页作为例子的。为了简单起见我不打算写代码。

朴素的同步阻塞

发起 HTTP 请求，等待响应，响应后，等待接收所有数据，接收完成后，一篇网页就下好了。这样干十次，就把事情干完了，但是是最慢的。

多进程

开多个进程干上面一次下载要干的事情，每个进程下不同的网页。这样其实没有本质的区别，如果 CPU 的核数和进程数一样多的话，实际上还是有大量的时间被浪费了。（如果 CPU 的核数少的话可以通过换进程来让 IO 和运算重叠。）

多线程

开多个线程干上面一次下载要干的事情，每个线程下不同的网页。这个和多进程的区别在于：

  1. 对 OS 来说，多线程的切换开销更小。
  2. 然而由于 GIL 锁的存在，多线程并行是不可能的。
  3. 不过，某个线程如果碰到了阻塞操作的话，会运行另一个线程。

总之，Python 中如果是计算密集型的，用多进程；IO 密集型的，用多线程。

轮询非阻塞

如果发起 HTTP 这个接口有非阻塞的版本，那么可以调用之后开始轮询，一般会通过更新了某个状态来知道事情干完了没有。不过如果没有更好的事情可以干并且只进行了一个调用的话，这样做的时间和同步阻塞几乎没差别。如果发了多个请求，然后轮着轮询的话，效率会好一点。

回调

回调说的是在发起请求的时候顺带传一个函数，请求处理完毕之后调用这个函数。具体怎么实现，操作系统将 IO 操作进行了封装，比如 IO 操作完成是一个 事件，当程序把所有的准备工作都干完了之后（调用了一堆异步的接口等着回调）会开启一个 事件循环，这个事件循环会不断取出事件进行处理，比如抽到一个 IO 操作完成的事件就会调用对应的回调函数。如果没有新的事件的话就会阻塞。操作系统的这种机制称为 select。（可以通过搜索 epoll/kqueue 来获得更多信息。）

使用回调时，只用一个线程就能搞定所有这一切。

回调的缺点：主要是层次过多（回调套回调）时结构复杂，此外回调中要保留这个回调要用到的一些状态；另外，回调一旦出错很难处理。这些问题让回调不是那么好用。

协程

所谓线协程，是一组互相协作的程序单元（一个例子是两个会相互调度（而不是调用）的函数）。这样每个程序单元维护自己的状态是容易的，并且它们之间的切换不用经过 OS 内核，代价很小。厉害的是它解决掉了前述中回调的诸多缺点。然而只要使用异步，总是不可避免回调的，不过协程可以让回调对程序员透明。

Python 中存在生成器，生成器可以保存自己的状态，可以暂时暂停执行然后恢复，这不依赖于线程或进程的切换。于是可以使用生成器来做协程。

定义以下几个类：

  • Future：这个类存放某个操作未来的结果，一旦结果被得到，就会调用事先加入其中的回调函数。
  • Task：这个类存放一个协程（本质上，是一个生成器），同时它有一个步进函数，步进函数需要传入一个 Future，其功能如下：
    • 将这个 Future 的结果传入生成器，并得到生成器的下一个生成结果。生成器的下一个生成结果经过设计，正好是一个空的 Future。
    • 为刚刚得到的这个 Future 添加一个“完成回调”，具体就是把自己传给自己这个 Task 的步进函数。
      另外，Task 的构造函数需要传入自己要维护的协程，并且会将这个协程运行到第一次 yield 的地方（通过调用步进实现）。

然后定义那些协程（生成器的时候）注意两点：

  • 每次 yield 返回一个空的 Future
  • 当要进行某个会阻塞的操作时，先构建一个空 Future，然后使用其非阻塞版本，将其完成的回调设置为“填充刚刚构造的空 Future”，然后 yield 出去。

首先曾经构造过了一些 Task，这些 Task 中维护的协程应该已经运行到了各自第一次 yield 的地方，并且它们内部都有个空的 Future。然后，假设某个阻塞操作终于完成了，此时事件循环捕捉到这个信号，运行了其上的回调函数。这个回调是什么，是“填充刚刚构造的空 Future 的结果”，因此会触发 Future 被“完成回调”，而这个“完成回调”是之前运行步进时添加的“把自己传给自己这个 Task 的步进函数”，因此现在的情况是，返回值放在了 Future 中，正在运行的是 Task 的步进，步进会“将这个 Future 的结果传入生成器”，于是协程中拿到了返回值，进行适当操作后，再次调用某阻塞函数然后 yield 出来一个空 Future，回到了步进函数中，步进函数又“为刚刚得到的这个 Future 添加一个‘完成回调’，具体就是把自己传给自己这个 Task 的步进函数。”，这一步进行完之后回到了事件循环。如此反复，直到事情干完（细节不写了）。
