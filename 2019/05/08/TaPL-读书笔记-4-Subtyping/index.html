TaPL 读书笔记 4 Subtyping

发表于 2019-05-08 | 更新于 2019-05-28

笔记对应原书 15-19 章。

上一部分（Simple Types）的重点回顾：

  • 使用 类型系统 可以 静态地 检查出一部分可能会出错的程序，而我们希望运行过程中会 stuck 的程序能够被拒绝。
  • 语言的定义会包含一系列的 typing rules，根据这些 typing rules 能够推断出一些 terms 的类型，这些 terms 是 well typed terms。
  • 类型系统依据 progress 性质 和 preservation 性质 来保证程序不可能 stuck，但不一定保证程序一定会终止。合格的类型系统必须保证这两个规则成立：
    • Progress：一个 well typed term 要么是 value，要么就能继续求值。
    • Preservation：一个 well typed term 如果求值了，那么得到的 term 还是 well typed 的。
  • 如果每个 well typed term 的类型唯一，则类型系统有 uniqueness 性质。
  • 如果每个 well typed term 求值必定终止，则类型系统有 normalizable 性质。
  • 我们可以给无类型 lambda 演算加入 函数类型，但是这样的演算系统没有 well typed terms。
  • 我们可以给带有一些简单类型的 lambda 演算加入各种扩展，实现 let binding、tuple、record、variants、语言内建递归等。
  • 在保证类型安全的情况下，我们可以给简单类型 lambda 演算加入 引用 和 异常 的扩展。

在这一部分的笔记开始前插一句，类型系统分为 nominal 和 structural 两种，差异是类型相同时如何比较。可以定义两个除了名字不同，其它都一样的类型，如果类型系统认为两个类型是同一个类型，这种类型系统就是 structural 的（比如 Haskell），否则就是 nominal 的（比如 C++、Java）。

我发现用多了多重标题后，一篇博客会充斥着无聊的内容，所以这次我尝试不写多重标题。

前面我们为无类型 lambda 演算引入了一个简单的类型系统，它确实实现了目的——拒绝掉会 stuck 的程序。然而这个系统在一些情况下用起来很麻烦，表现力不够强，如果一个函数要接受某种记录类型，那即使传入了多了几个域的“兼容”的记录类型也是不行的，然而这个程序明显是正确的。解决这个问题的办法是引入 subtyping，对于两个类型 S 和 T 用 S <: T 表示 S 是 T 的子类型，意思是 所有 T 可以出现的地方都可以用 S 进行替换，任何类型为 S 的 term 的类型也为 T，就像 OOP 里面的继承那样。显然 subtyping 关系是自反、传递、反对称的（构成偏序集）。

可以证明引入了 subtyping 后类型系统还是可以发挥作用，依然满足 progress 和 preservation 性质。另外，subtyping 并没有增强类型系统的功能，只是提高了它的表达能力。通过一种叫做 coercion semantics 的东西，可以对给定的 subtyping 规则，构造出一组将其翻译到不带 subtyping 的类型系统的语言的规则，例如将有 subtyping 的 lambda 演算翻译到简单类型 lambda 演算，所以它们的表达能力是一样的。

为了便于讨论一般还会引入两个类型 Top 和 Bot。其中所有类型都是 Top 的子类型（这也意味着如果你知道一个值的类型是 Top，相当于你什么都不知道）；Bot 是所有类型的子类型（它没有 closed form term，一般用于表示不会返回的计算过程，例如不会终止的过程或者一定会抛出异常之类的）。在一些 OOP 语言中所有的类都有个父类 Object，这就相当于 Top。

另外还可以引入 intersection type 和 union type 的概念。由于类型可以看作是 term 的集合，subtyping 关系可以构造偏序集，因此可以定义两个类型的 meet 是它们的 intersection type，包括了同时属于这两个类型的 term（即集合的交）；两个类型的 join 是它们的 union type，是 terms 的集合的并。Union type 和 variant 的区别在于前者不需要 tag（当然也就没法区分到底是哪个类型）。关于 intersection type 有一些定理：

  • 对于引用，它不是逆变的也不是协变的，而是 invariant 的。如果 T 的引用是 R[T]，那么从 S <: T 不能得到任何 R[S] 和 R[T] 的关系。我们可以将引用操作拆分为 读 和 写 两部分。如果只让读不让写，这种类型称为 Source，它是协变的；如果只让写不让读，这种类型称为 Sink，它是逆变的。引用 R[T] 既是 Source[T] 也是 Sink[T]，所以它只能是 invariant 的。

书中后面的内容介绍的是如何使用 lambda 演算实现 OOP（带有状态的对象；封装、继承、多态等特征的实现等），这个叫做 object encoding。当然为了实现状态，不可避免地要使用引用来对状态进行维护。实现中的一些关键思路（详细内容看书）：

  1. 使用一个记录存放所有的状态，不存放任何方法。这个记录叫做 representation record。
  2. 搞一个函数，它接受一个 representation record，返回一个记录。这个记录中都是方法（叫它 object record 好了），并且访问了 representation record。现在原始的 representation record 就被 封装 起来没法访问了，只能访问对应的 object record。这一步实现了最简单的 OOP。
  3. 为了支持 继承，在定义 object record 的时候可以先构造基类的 object record，将其绑定到一个叫做 super 的变量上，然后在定义子类的 object record 时可以使用 super。
  4. 为了支持 多态，需要实现 self 的 late binding。定义 object record 的函数除了要接受一个 representation record 外还有接受一个 self 参数，进行 late binding。这么操作之后就可以实现类似虚函数 dynamic dispatch 的效果。
  5. 最后为了支持调用对象内部的函数需要用到不动点的手段，用了这个手段后计算可能无法终止，需要用 thunk 进行封装。

这只是说明 lambda 演算可以对 OOP 进行编码，实际中证明 OOP 的类型系统的性质时不可能用这么复杂的方法。为了证明 OOP 类型系统的性质，可以对这个语言进行不同于 lambda 演算的建模，例如作者提出的 Featherweight Java。这种做法会对现有的 OOP 语言大幅度简化，从头开始定义 evaluation rules、typing rules 和 subtyping rules，然后证明类型系统的 progress 和 preservation 性质从而证明类型系统的正确性。
