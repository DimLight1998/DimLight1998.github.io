Parser Generator 与 Parser Combinator

Parser 有两种——Generator 和 combinator。

  • Generator 接受语法规则，然后生成语法分析程序。例如 ANTLR 就是典型的 parser generator。

  • Combinator 将一些小的语法分析函数组合起来，形成功能更复杂的语法分析函数。

Happy 和 Parsec 是 Haskell 的两个包，前者是 parser generator，后者是 parser combinator。

作为例子，这篇文章会使用两种不同的策略对 json 文件进行语法分析。

Parsec - Parser Combinator

Parsec 使用 parser combinator 的策略：每个函数处理一件小事，然后通过函数组合让它的功能更加复杂。

Parsec 最低可以到字符粒度的控制，因此既可以进行词法分析也可以进行语法分析。Parsec 库中提供了大量可以直接使用的函数，只需要将这些函数组合起来即可，并且这些函数的语义比较明确。

处理数字

使用 Parsec 时，所有的做解析的函数的签名都应为 GenParser Char st X，其中 X 按照自己需要决定。

Json 中数字的正则表达式是 -?(0|[1-9][0-9]*)(\.[0-9]+)?([eE][+-]?[0-9]+)?。下面首先使用 integer、fraction 和 exponential 分别处理整数部分、小数部分和指数部分，它们处理的结果都“返回”字符串。然后使用 number 将它们结合起来最终返回一个 JValue。

以 integer 为例，首先 string "0" 会尝试从字符串流中匹配 "0"，一旦成功，立即返回；否则会开始匹配 <|> 后的内容（<|> 是 Alternative 的一个运算符）。oneOf 会从给定的多个字符中进行匹配，digit 会尝试匹配一个数字，many 会进行多次匹配（可以是 0 次，而 many1 不允许是 0 次）。

需要注意对于 <|> 来说，如果前面已经匹配了一部分然后失败了，那么匹配后面时前面已匹配部分已经被消耗了。可使用 try 解决这个问题。

choice 会从一个列表的匹配函数中依次选择直到成功，option 会在不成功时返回一个 fallback 值。

处理字符串

count 可以指定匹配次数；noneOf 相当于反向的 oneOf。

使用 choice 时，也会出现匹配一部分后失败导致字符被消耗的问题，因此这里用 map try 来解决。应该有更好的办法。

核心的处理

space 可以匹配空白字符（因为没有 tokenize 的过程，所以即使是语法解析阶段也要手动处理空白字符）；sepBy 可以匹配模式 A 被模式 B 分割的模式（endBy 则匹配模式 A 总以模式 B 结尾的模式）。
