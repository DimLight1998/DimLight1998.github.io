SQL 语言与数据库完整性和安全性

数据库是概念，SQL 是实现。二者不是同一个抽象层上的东西，看的时候注意这一点。

数据库完整性概念及完整性约束规则

数据库完整性

DBMS 应当保证数据库始终具有某种性质，这种性质一般和正确性、有效性和一致性有关。

完整性涉及到语义上的完整性和数据库实现上的完整性。

为什么会引发完整性问题

语义相关：错误数据输入、操作失误

实现相关：程序处理失误

DBMS 应当防止这些错误导致的语义不合理现象。

如何保证完整性

DBA 会通过 DDL 定义语义完整性。用户修改数据时，DBMS 仅在通过检查后允许修改。

完整性约束条件是四元组 $(O, P, A, R)$，分量含义分别为数据集合（约束对象）、谓词条件（什么样的约束）、触发条件（何时进行检查）和响应动作（不满足时触发的动作）。

数据库完整性分类

按约束对象分类

  • 域完整性：施加于某一列的约束。只给了一列数据也能判定某记录是否符合条件。例如要求某列的值必须大于 10。
  • 关系完整性：施加于记录上的语速。例如要求某记录的某两个域的和必须大于 10。

按约束来源分类

  • 结构约束：来自模型的约束，例如主键不为空、外键必须是候选键等。
  • 内容约束：用户定义的语义约束。

按约束状态

  • 静态约束：数据在任意时刻都应该满足的约束。
  • 动态约束：数据在进行修改时应当满足的约束。例如约束某一列的值只能升不能降。

SQL 表完整性与列完整性

SQL 使用 schema 进行静态约束，使用触发器进行动态约束。

定义 schema 时可以对列进行约束，也可以对表进行约束。

  • 对列可以进行的约束有 NOT NULL、UNIQUE、PRIMARY KEY、CHECK(condition)、REFERENCES tableName [(colName)] 和 ON DELETE <CASCADE | SET NULL>。可以使用 CONSTRAINT constraintName 为约束命名，方便日后撤销。其中 REFERENCES 用来设置外键。
  • 对表进行的约束实际上是对多列共同进行约束，可以使用的约束有 UNIQUE（多列构成候选键）、PRIMARY KEY（多列构成主键）、CHECK（多列共同满足条件）、FOREIGN KEY (colName1, colName2, ...) REFERENCES tableName(colName1, colName2, ...) 和 ON DELETE CASCADE。可以使用 CONSTRAINT constraintName 为约束命名，方便日后撤销。

列约束是表约束的特例。上述 CHECK 后的条件和 WHERE 子句后的条件格式相同，可以写出复杂的条件。

在修改表时可以使用 ALTER TABLE。可以追加列、删除列、修改列定义。如果想要追加列约束可以通过修改列定义来完成。

使用 ADD CONSTRAINT constraintName 来增加约束、DROP CONSTRAINT constraintName 来删除约束、DROP PRIMARY KEY 删除主键。

数据库安全性的概念

前面几节记录的是数据库的 完整性，下面几节记录的是 安全性。

数据库的安全性体现为防止未授权用户对数据库的访问与破坏。安全性的分类有

  • 自主安全性：权限可以在用户之间传递，用户自主管理安全性。
  • 强制安全性：对数据和用户进行分类，不同用户能够访问不同数据。
  • 推断控制机制：防止用户通过多次查询结果推断出某些个体的信息。
  • 数据加密存储。

DBA 是 DBMS 中的超级用户，可以创建其它用户并授权。

自主安全性机制

自主安全性是通常的安全性实现。

DBA 会对用户授权，而用户也可以将自己的权限给别人。

权限是一个四元组，其组成：

  • 请求主体：谁拥有这个权限。
  • 访问对象：可以是数据库、表、记录、域。
  • 权利：创建、插入、删除、修改、查询。
  • 谓词：拥有权利需要满足的条件（比如只能访问某个域大于 100 的记录）。

若干条上述权限的记录形成了一张访问规则表（逻辑上的，实现上不一定存储了这张表），DBMS 根据这张表进行安全性检查。

两种自主安全性控制

存储矩阵

将每个主体作为一行，可能的访问对象作为一列，使用矩阵进行存储。矩阵元素为该主体对访问对象的权利。

视图

通过在表上创建视图来进行安全性控制。创建时视图可以指定条件，因此可以做得比较复杂。此外还能反映权限中的谓词。

让不同主体能够访问不同的视图即可。

二者不是对立的，可以同时使用。建立了视图后，视图也会作为一个访问对象。视图可以用来实现权限中的谓词，这点比较重要。

SQL 安全性控制

权限级别

数据库安全性属于 DCL 范畴。

权限被分为三个层次：

  • 读权限：只能读不能更新。
  • 对记录的更新：可以插入、修改、删除记录。
  • 对关系的更新：可以创建、修改、删除关系。这是对记录的集合的操作

一般来说高层次会包含低层次的权限。例如一个用户创建了某个关系后就可以向关系中插入记录。

自主安全性控制的问题

授权传播范围

有些 SQL 实现控制了权限传播时的范围。权限的传播会构成一张以用户为结点的有向图，每次授权都会增加一条从授权者指向被授权者的边，范围指的是图上任意一条路径的长度（授权深度）和任意一个结点的出度（授权广度）。通过限制这两个量才能保证被授权者数量被控制在一个可控的范围内。

被多个人授权

如果一个用户被多个用户授予了相同权限，当其中一个用户收回权限时，该用户仍然具有相应权限，因为其他人也授予了他这个权限。

除非所有人都要将对应的权限收回，否则这个用户就一直有这个权限。

强制安全性机制

强制安全性通过将数据分级，同时将用户分级来强制性地实现不同用户对不同数据的访问。

访问规则：

  • 用户不能读取分级比自己高的数据。
  • 用户不能修改分级比自己低的数据。

其中第二条是因为，一旦用户修改了这些数据，这些数据的安全性级别就会上升，导致原来可以读取这些数据的用户现在不能读取了，这是我们不希望发生的。

在采用这种机制时，定义 schema 时就要指定关系、属性和元组的分级。
