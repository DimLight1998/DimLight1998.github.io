数据库索引技术

索引的概念和作用

什么是索引及其特点

索引用来在不用遍历所有记录的情况下快速定位所需记录在整个数据库中的位置。它是一种辅助结构，存在与否不改变数据表本身的组织。

索引由一系列 索引项 构成，每个索引项包括 索引字段（记录的被索引字段的值）和 行指针（指向记录的位置）。

存储索引项的文件称为 索引文件，数据库本身称为 主文件。

特点：

  • 一张表上可以对多个不同属性（及其集合）建立多个不同的索引文件。
  • 索引文件比表要小很多，一般可以全部装入内存。
  • 对主文件的更新应当同步更新索引文件。

索引文件的组织

索引文件有 排序索引 和 散列索引 两种组织方式，依据都是其中的索引字段。

对索引文件衡量标准

  • 时间：访问、插入、删除、更新索引文件对应的时间。
  • 空间：引入索引后增加的空间。
  • 便利程度：能否支持单一限制的查询；能否支持范围查询。

关于索引的一些概念

  • 排序码：主文件进行排序存储时用到的属性组
  • 索引码：索引字段，不一定是唯一的
  • 搜索码：用于在主文件中查找记录时用到的属性组

SQL 中索引的创建和使用

SQL 在用户定义主键后会自动创建 主索引。用户也可以手动创建索引或者撤销。SQL 会自动维护索引文件和主文件的一致性。

稀疏索引与稠密索引

如果索引文件中包括了被索引字段的所有值，这样的索引策略称为 稠密索引，反之称为 稀疏索引。注意稠密索引的索引项数量不一定等于表中记录的数量，因为索引码可能有重复。

当使用 稀疏索引 时，为了找到需要的记录，应当先使用稀疏索引找到比所需记索引码小的最大的记录，然后往后顺序查找。

当使用 稠密索引 时，为了找到需要的记录只要看一下稠密索引中是否有对应的记录的索引码即可。没有的话就表示表中没有。

主文件在索引码上的排序

在使用稠密索引时，如果表中有被索引的字段相同的记录：

  • 如果索引项不允许重复，那么可以考虑将主文件按照索引码进行排序，索引项指向排序后与之相等的第一条记录。
  • 如果索引项允许重复，则主文件不用进行排序。可以建立和记录数目相同的索引项。
  • 如果索引项不允许重复，主文件不能进行排序，那么要引入一个中间层：每个索引项指向一个链表，链表中的每个元素再次指向一条记录。

主索引与辅助索引

主文件是按照存储块进行存储的。前面说到的索引都是对每条记录进行索引，但是这不是必须的。实际上可以只对主文件占用的每个存储块进行索引，每个索引项对应一个索引块，指针指向索引块的起始位置对应的记录（称为 锚记录）。

这种策略称为 主索引，它是一种稀疏索引。主索引使用的索引码一般是表的排序码，很多时候还是主码。主索引只能有一个。

辅助索引 是索引码不是排序码的索引，它是稠密索引。有时候使用辅助索引的效率会很高。尽管实现上可以将辅助索引指向块，但是一般实现为指向记录，这样更快。

聚簇索引与倒排索引

聚簇索引与非聚簇索引

如果索引中临近的记录在表中也是临近的，这样的索引称为 聚簇索引，否则为 非聚簇索引。

当使用聚簇索引时索引码一般是排序码。一张表只能有一个聚簇索引。主索引是聚簇索引，辅助索引是非聚簇索引。

聚簇索引可以用来修改表中的数据，例如要插入一条记录的时候就是通过聚簇索引完成的。但是非聚簇索引只能用来查询，不能用来修改。

倒排索引

倒排索引是来自文档检索的概念，它希望的是优化给定关键字时查找对应文档的效率。

  • 正排索引：存储每个文档对应哪些关键字
  • 倒排索引：存储每个关键字对应哪些文档

B+ 树索引

什么是 B+ 树索引

B+ 树是一种 多级索引：索引还可能是对其它索引的索引。

B+ 树的每个结点存放 n 个值和 n + 1 个指针，一个结点应当正好放入一个磁盘块当中。

B+ 树中非叶结点的所有指针都指向下一层的结点，叶结点的指针中，最后一个指针指向下一个叶结点，其它指针指向表中记录位置或者对应的块的位置。因此使用 B+ 树进行索引时，可以进行范围的查询。

B+ 树索引不要求是索引码是排序码，尽管它可以进行范围查询。

使用 B+ 树进行索引

B+ 树可以建立不同种类的索引，根据需求而定。

  • 索引码是主键。此时可以建立稠密索引，最终指向记录。主文件可以不按照主键排序。
  • 索引码是主键。此时可以建立稀疏索引，最终指向块。主文件需要按照主键排序。
  • 索引码不是候选键，但是主文件按照索引码排序了。此时可以建立稠密索引。指针最终指向具有相同索引码的第一条记录。
  • 索引码不是候选键，主文件也没有按照索引码排序。此时可以建立稠密索引，但是要求 B+ 树结点中存储边界的那些值允许重复，就像这样：

可扩展散列索引和线性散列索引

散列分为 静态散列索引 和 动态散列索引。

静态散列索引在每个桶溢出时会用链表连接一些“溢出桶”来保证可用，但是直接的桶的数量是固定的。

动态散列索引中，直接的桶的数量不是固定的，可能随着使用越来越多。引入的问题就是，原来已经被散列的数据是否要重新被散列？

可扩展散列索引

可扩展散列索引通过引入一个中间层部分程度上解决了增加桶的数目后需要重新计算各元素哈希值的问题。中间层是一个数组，每个元素指向一个桶，但是不同元素可以共享桶。增加桶时数组可能会扩大。详细的内容可以见 https://en.wikipedia.org/wiki/Extendible_hashing。

线性散列索引

线性散列索引对可扩展散列索引进行了优化，它也有一个中间层，但是每次要扩张中间层的时候不是指数扩展的，而是线性扩展的。线性散列索引允许溢出。详细的内容可以见 https://en.wikipedia.org/wiki/Linear_hashing。
