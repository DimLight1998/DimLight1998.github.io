数据库系统的结构抽象与演变

数据库系统的标准结构

三层次抽象

数据的存储可以看做有三个抽象层次：

  • 外部层次（用户层次）：DBAP 可以看到的层次，一般是所有数据的一部分。
  • 概念层次（逻辑层次）：DBMS 使用到的层次，包括对应的约束和关联。
  • 内部层次（物理层次）：数据在介质上的存储，包括路径、索引等。

模式与视图

  • 模式（Schema）：对数据库中的数据的结构化描述。
  • 视图（View）（又称数据）：通过某种表现形式表现出的数据库中的数据。它是一种表现。

将三层次抽象和模式视图结合，排列组合一下可以得到六个概念：内部模式、内部视图、概念模式、概念视图、外部模式、外部视图。当然知道这几个概念并没有什么用。

在不指明抽象层次的情况下，模式指的是概念模式，视图指的是外部视图。

两层映像

两层映像指的是不同抽象层次之间的 模式 的映射：

  • C-E 映像：将概念模式映射到外部模式，这么做是为了方便用户观察和使用。
  • I-C 映像：将内部模式映射到概念模式，这么做是在保证计算机方便处理的前提下提供足够的抽象性。

DBMS 应该能让用户自己定义三个层次的模式，然后自动生成对应的两层映像。

一般只会有一个概念模式，但是可以有多个内部模式和外部模式。

逻辑数据独立性与物理数据独立性

抽象层次使得一层发生变动时其它层次可以几乎不用修改。各层有一定的 独立性。

  • 逻辑数据独立性 指的是在概念模式发生变动时，外部模式不用进行修改，只要修改 C-E 映射即可。
  • 物理数据独立性 指的是在内部模式发生变动时，概念模式不用进行修改，只要修改 I-C 映射即可。

什么是数据模型

数据模型

有三个概念，数据、模式和数据模型。下面的描述中将三者加粗。

数据 的模式称为 模式，模式 描述了 数据 应当如何进行组织，模式 是对 数据 的结构的抽象。

模式 的模式称为 数据模型，数据模型 描述了 模式 应当如何进行组织，数据模型 是对 模式 的结构的抽象。

数据模型的三个经典实例

三个经典的数据模型分别是关系模型、层次模型和网状模型。

关系模型是很多数据库采用的数据模型，使用表组织数据。但是并非所有的数据库都是关系型的。

层次模型和网状模型分别使用有根树和有向图图来描述 模式，这些树和图中，每个结点称为 实体型、边称为 系型。每个实体型是对一个实体应当如何描述的模式，在存储数据时，多个实体按照实体型制定的模式进行存储，构成一个集合。不同类型的实体的集合之间通过有向边进行连接。

数据库系统的演变之一

简要发展史

数据库的研究从 1959 年左右开始，在 1976 年左右开始成熟。早期的数据库很多不是关系型的，在成熟时关系型数据库和对应的理论开始迅速进步。

重要演变经历

从文件系统到数据库

早期数据管理使用操作系统提供的文件系统直接进行管理，但是此时数据和程序的耦合性严重。因此过渡到了数据库。数据库的引入使得数据和程序解耦，提高了数据的共享程度。

从层次模型和网状模型到关系模型

早期数据库有些是层次模型和网状模型的，实现要用到指针（指向关系要用户手动建立），很复杂而且难以对记录的集合进行操作。关系模型理论成熟并且容易对记录的集合进行操作，并且容易实现非过程化的查询。

使用了层次和网状模型的数据库称为第一代数据库，使用了关系模型的数据库称为第二代数据库。

数据库系统的演变之二

重要演变经历（续）

关系数据库到对象关系数据库和面向对象数据库

  • 复合属性：关系型数据库中，每个域都是原子化的，不可继续分割。但是有时候希望一个域下面还有子域，可以进行分割，这种域称为复合属性。
  • 多值属性：关系型数据库中，每个记录的一个域只有一个值。但是有时候希望它能够有多个值，这种域称为多值属性。

通过引入 OOP 中的一些思路可以解决上述问题。引入列对象和行对象的概念解决复合属性的问题，引入聚集对象和结构对象的概念解决多值属性的问题。此时数据库中每个值都是一个对象。引入这些玩意后关系数据库就变成了对象关系数据库，并且还能支持多态等 OOP 概念。

其它

  • XML 数据库：一种半结构化数据库，数据和数据的语义合并存储和处理。
  • 从多种数据库能够进行开放式互联的变化。多种不同厂商的数据库通过同样的标准可以互相访问。
  • 其它各种新型数据库。

回顾本讲学习了什么

  • 数据、模式、数据模型。
  • 外模式、概念模式、内模式。
  • E-C 映像、C-I 映像。
  • 逻辑独立性与物理独立性：抽象层次引入导致的解耦。
  • 网状模型、层次模型、关系模型。
  • 各种不同的数据库。
