Building Compound Types

基础类型和复合类型

基础类型是“原子”的类型，无法继续分解。

复合类型例如列表、元组和 Option。

构造复合类型的通用方法

复合类型都可以由三种方式组合地构造出：

  • Each of：包括全部的可选类型
  • One of：包含若干个可选类型之一
  • Self reference：包含自己这个类型

例如元组就是 Each of 形式的，Option 就是 One of 形式的，列表是三种形式的组合：一个 int list 可以为空（One of），也可以为一个 int 加上一个 int list（Self reference 和 One of）。

Records

记录类型本质上是有名字的元组。

  • 构造：val r = {foo = 123, bar = false} 。

  • 访问：#foo r = 123, #bar r = false。

记录的键是无序的。

记录和元组的区别在于一个是靠名称访问，一个是靠位置访问。

一个事实是很多语言中的函数调用是二者的结合：调用者使用位置访问传递参数，被调用者使用名称访问读取参数。有些语言允许不这样做。

Tuples as Syntactic Sugar

元组本质上是记录的语法糖。可以认为 ML 中没有元组，只有记录。以下两个绑定是等价的。

记录的键可以是数字。当键不是 1 到 n 时，会成为普通记录。

Useful Datatypes

Datatype 可以用来实现枚举（并且元素自己还可以带有属性）。

Datatype 相当于一种 one of 的结构，记录相当于一种 each of 的结构。两者不要混淆使用。

Lists and Options are Datatypes

列表和 Option 上是 datatype，通过使用模式匹配，可以避免掉使用 isSome、valOf、null、hd 以及 tl，因为这些函数实际上都可以使用模式匹配等价地搞出来。

例如列表的构造函数实际上是 [] 和 ::。

这些函数存在的意义是有时候会便利，并且有时候需要将函数作为参数传递的时候有用。

可以认为列表和 Option 是可以自己造出来的，语言不一定要提供。

Polymorphic and Equality Types

类型的广泛性

如果一个复合类型中存在形如 'a 的类型变量，这种变量可以替换为一个具体的类型或者其它类型变量。一个复合类型 $T_1$ 比另一个复合类型 $T_2$ 更加广泛的判定是，如果将 $T_1$ 中某个类型变量的所有出现之处全部替换为另一个类型变量或者具体的类型能够得到 $T_2$，则 $T_1$ 比 $T_2$ 更加广泛。

Equality 类型

形如 ''a 的类型表示这个类型不仅是类型变量，而且可以使用 = 来判定相等性。

Nested Patterns Precisely

这节对模式匹配能否成功做了递归的定义。

  • 如果模式是 $p$，值是 $v$，可以匹配，会引入绑定 $p = v$。
  • 如果模式是 $_$，值是 $v$，可以匹配，不会引入绑定。
  • 如果模式是 $(p_1, p_2, \dots, p_n)$，值是 $(v_1, v_2, \dots, v_n)$，可以匹配并引入对应的绑定。
  • 如果模式是 $C\ p$，其中 $C$ 是一个构造函数，并且值是 $C\ v$，可以匹配并且引入绑定 $p = v$。

上面的准则中的模式不一定是变量，也可以是复合的模式，这样一来整个准则可以递归应用下去。

Tail Recursion

如果 $f$ 调用 $g$，但是 $f$ 的结果就是 $g$ 的结果而没有做任何变化和读取，就只是原封不动地传递回去了，这个情况下就是尾递归（不一定是在调用自己）。

第一个的情况下，因为 fact 调 fact 之后对返回值还要加以利用，所以不是尾递归。这种情况下，每次调用都会用掉一个栈帧，比使用循环效率低很多。

第二个的情况下，所有使用了 aux 的地方都是以尾递归的形式使用的。凡是这种情况，ML 会在运行时将新的 aux 的栈帧直接替换掉调用 aux 的那个函数的栈帧（毕竟这个函数做的唯一的事就是继续返回 aux 的返回值），这样一来栈空间占用就可以保持在一个较小的水平。只要使用了尾递归，即使是无限递归也不会导致栈爆掉。

Accumulators for Tail Recursion

有一个经验性的法则可以把递归转换成尾递归。一般来说需要一个辅助函数和一个 accumulator 表示当前完成的工作，这个辅助函数的参数是原函数的参数加上额外的这个 accumulator。辅助函数的实现中，如果发现原函数对应的那些参数是递归基的情形就返回 accumulator，否则计算新的参数并直接调用辅助函数（就是说所有的计算最好在单个表达式中完成，并且表达式就是新被调用辅助函数的参数）。

Perspective on Tail Recursion

并非所有的递归都 适合 转换成尾递归。理论上，所有的递归确实可以写成尾递归，但是 accumulator 占用的空间可能和不适用尾递归时栈帧的大小差不多，这相当于自己实现了一个栈管理器。

什么调用是一个尾调用是有严格的递归定义的。每个表达式都有一个 尾区域。例如：

  • $\texttt{fun f p = e}$ 中的尾区域包括 $\texttt{e}$，不包括其它。
  • $\texttt{if e1 then e2 else e3}$ 中的尾区域包括 $\texttt{e2}$ 和 $\texttt{e3}$，不包括其它。
  • $\texttt{let b1 b2 ... bn in e end}$ 中的尾区域包括 $\texttt{e}$，不包括其它。

给定了各个表达式的尾区域中，有一个规则：如果一个表达式不在尾区域中，那么它的任意子表达式也在尾区域中。递归地应用这些定义可以得到相对于每个表达式的尾区域。

尾调用指的是位于尾区域中的调用。（这个说法中的尾区域是相对于某个表达式而言的，因为如果再深入看一点进入更小的子表达式，这个尾区域中的一个子集才是尾区域。）
