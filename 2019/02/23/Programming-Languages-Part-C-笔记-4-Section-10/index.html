Section 10

Subtyping From the Beginning

接下来的几节中使用了作者自己定义的语言。语言中的记录类型和 ML 很类似，并且是 mutable 的。语法：

  • 字面量：e = {f1=e1, f2=e2, ..., fn=en}
  • 读写：e.f1 = e1
  • 类型标注：{f1: t1, f2: t2, ..., fn: tn}

这个语言可以建立简单的类型系统，该系统的规则和直觉一致。

The Subtype Relation

为了引入子类型，加入两条规则：

  • 将 t1 为 t2 的子类型记作 t1 <: t2
  • 如果 e 的类型是 t1，t1 <: t2，那么 e 也具有 t2 类型

正确的子类型定义应该保证凡是能够出现 e2 实例的地方，必然也可以使用 e1 进行替换。将 程序访问或修改一个不存在于记录中的域 作为一个要拒绝的性质，在引入子类型前，类型系统是可靠的。为了保证引入后类型系统依然可靠，需要继续满足前述性质。为此定义如下的子类型规则：

  • 如果类型 t1 具有类型 t2 所有的域（并且对应的类型一致），那么 t1 <: t2
  • 如果类型 t1 和 t2 域相同（并且对应的类型一致），但是排序不同，那么 t1 <: t2
  • 自反性：对任意 t1，有 t1 <: t1
  • 传递性：对任意 t1，t2，t3，如果 t1 <: t2，t2 <: t3，则 t1 <: t3

按照这个定义可以继续保证类型系统的可靠性，因为对记录的操作只有读和写，而对某个记录的读和写对其子类型也适用。

Depth Subtyping

假设我们为上述的子类型引入新的定义：

  • 如果 ta <: tb，那么 {f1: t1, ..., f: ta, ..., fn: tn} <: {f1: t1, ..., f: tb, ..., fn: tn}

按此定义，凡是使用了 circle = {center={x=3.0, y=4.0}, color='red'} 的地方，都可以使用 sphere = {center={x=3.0, y=4.0, z=5.0}, color='red'} 进行代替。

Java 为了保证：

  • 对于 foo.bar 的使用，永远不会出现 foo 没有 bar 这个域的情况。
  • 数组读取 foo[bar] 中，如果 foo 的类型是 t1[]，那么读取出来的内容一定是 t1 或其子类型。

对上述情况，Java 在数组写入时会做动态检查：如果有代码 foo[bar] = bazz，除了静态检查外，如果 foo 的动态类型是 t1[]，而 bazz 的动态类型不是 t1 的子类型，就会抛出 ArrayStoreException 异常。

关于 null：null 是所有类型的子类型，虽然理论上讲，应该把所有的类型都作为 null 的子类型，但是实现上往往相反。使用 null 会带来运行时性能的损失。

另一种设计是像 ML 那样引入 Option，而不是用 null。

Function Subtyping

从维基上抄的，假设有一个类型构造器可以把一个类型 t 构造成另一个类型 g[t]

  • covariant（协变）：若 t1 <: t2，则 g[t1] <: g[t2]
  • contravariant（逆变）：若 t1 <: t2，则 g[t1] >: g[t2]

对于函数而言，它的参数是逆变的，返回值是协变的，即如果 t1 <: t2 且 t3 <: t4，则

  • (t1 -> t3) <: (t1 -> t4)
  • (t2 -> t3) <: (t1 -> t3)

Subtyping for OOP

OOP 中的子类型和前面几节的原理类似。将 OOP 语言中的每个类当成一个类型，将继承关系当做是子类型关系。因为继承时会保留所有的成员，所以和之前看到的原理基本相同。

理论上，可以在子类中对父类的方法进行重写，并且保证参数逆变、返回值协变即可。实际上，在 Java 和 C# 中只支持了返回值协变的重写。如果对参数进行了逆变操作，那么会变成静态的重载，多了一个方法。

类和类型是两个不同的概念——类是对象行为的描述，子类继承了行为描述并且作了修改；类型是对接口的描述，子类型替换了其中的一些类型。但是大部分 OOP 语言都刻意混淆这两个概念，这种混淆是很方便的。

最后，一个有趣的事实是，OOP 语言中的方法可以看做是接受特殊参数 self 的函数。前面提到过函数的参数是逆变的，但是 self 是很特殊的参数，在讨论函数的子类型时，它要按照协变来处理！因为这个参数可以访问到类中成员，只有设计成协变的才能继续保证类型系统的可靠性。

Generics Versus Subtyping

泛型和子类型是两种不同的策略，适用于解决不同的问题。二者不是相同的。

泛型可以用来实现例如泛型高阶函数的功能。FP 和 OOP 中都有泛型的概念。在能够使用泛型的地方使用子类型效果可能不好。例如使用泛型实现容器时，如果使用了子类型，那么存入容器的时候还好，从容器中读取的时候就不知道元素的具体类型是什么，需要作动态类型转换。

子类型可以用来在需要提供某些信息的地方提供额外的信息。ML 中没有子类型的概念，但是可以模拟出来（传入任意参数，以及对应的 getter）。在能够使用子类型的地方使用泛型效果可能不好。例如写 GUI 程序时，需要的公共功能是能够显示在屏幕上。如果使用了泛型就失去了多态的使用。

Bounded Polymorphism

使用 bounded polymorphism 可以把泛型和子类型结合起来使用。

泛型：对于所有的类型 T。

子类型：对于 A 的子类型 B。

结合起来：对于 A 的所有任意子类型 T。
