Scala 是纯 OOP 的吗

值类型可以被 OOP 实现

纯 OOP 语言指的是这种语言中所有的东西都是基于类和对象的。Java 不是纯 OOP 语言，因为其中的 primitive types 不是基于类或者对象的。

Scala 是纯 OOP 的，尽管它也有像 Java 中的 primitive types，但是 逻辑上 这些类型都是用 OOP 实现的。在编译时，编译器会特殊地对待这些类型，将其转化为更加底层的类型以保证运行效率。

OOP 可以实现诸如 boolean 和 int 这样的类型，并且实现一个原生类型时不需要依赖这个类型，即值类型可以被 OOP 体系实现。比如，可以用 OOP 实现皮亚诺算术进而实现整数，尽管这么做效率很低，但是确实是可行的。

函数类型可以被 OOP 实现

函数实际上也可以被 OOP 实现。类型 A => B 实际上是类型 Function1[A, B]，其中 Function1 是一个 trait。

Scala 泛型类的协变逆变约束

对于类型 A 和 B 有 A <: B，如果有某个类型构造子 C，则下面三种情况必有一个成立：

  • covariant：C[A] <: C[B]
  • contravariant：C[A] >: C[B]
  • nonvariant：上面两个都不成立

Scala 中，声明泛型类的时候可以声明这个泛型类对于类型参数是协变的、逆变的还是不变的：

  • convariant：class C[+A] { ... }
  • contravariant：class C[-A] { ... }
  • nonvariant：class C[A] { ... }

求值策略

Call by Value 与 Call by Name

程序对函数调用的求值策略有 call by value 和 call by name 两种。

  • 前者在函数调用前先对参数求值，然后将参数的值代入函数体中。
  • 后者直接将参数的表达式代入到函数体中，然后对函数体求值。

两种方式各有优点。前者不会对参数重复求值，后者不会对不被使用的参数求值。根据 lambda 运算理论，只要整个求值过程中没有副作用，并且两个过程都能终止，那么最后的结果总是相同的。

另一个常被提及的是 call by reference。然而这是一种传递参数的策略，而不是求值策略。为了防止混淆，我把按值传递和按引用传递叫做 pass by value 与 pass by reference。

不同求值策略的终止性

根据相关理论，如果 call by value 可以终止，那么 call by name 也可以终止，反之不成立。但是一般的语言（包括 Scala）总是默认 call by value，因为这样效率更高。

Z3 学习笔记

发表于 2019-04-09 | 更新于 2019-04-16

Z3 学习笔记

Introduction

Z3 is a theorem prover which can be used to check the satisfiability of logical formulas over theories.

Basic Commands

A Z3 script is a sequence of commands defined by SMT-LIB 2.0 standard.

The syntax is like Racket.

  • echo to print something.
  • declare-const to declare a constant. (No value needed.)
  • declare-fun to declare a function. (No definition needed.)
  • assert to make assertions, which will be used to check satisfiability.
  • check-sat to check the satisfiability of all assertions.
  • get-model to get the assignment if all assertions are satisfiable.

The result of get-model uses define-fun to define functions.

An ite expression stands for an if-then-else branch.

Using Scopes

Z3 uses a global stack to store all declarations. This is useful if you want to prove multiple independent theories using some common declarations.

Use push to enter a new stack frame and use pop to exit the current stack frame.

Configuration

Use set-option to configure Z3. Use reset to remove all configurations and assertions.

Some parameters for set-option:

  • :print-success will toggle printing success for commands.
  • :produce-unsat-cores will toggle generation of unsat cores.
  • :produce-models will toggle model generation.
  • :produce-proofs will toggle proof generation.

Additional Commands

  • display to pretty print an expression.
  • simplify to simplify an expression. Many options are available.
  • define-sort to define a new type. The type can be parameterized.

You may need as as something like type annotations when dealing with parameterized types.

Propositional Logic

Use and, or, xor, not, => (for $\rightarrow$) and = (for $\leftrightarrow$) for logical connectives.

Satisfiability and Validity

If you want to prove $F$ is valid, you can ask Z3 to prove $\lnot F$ is unsatisfiable.

Uninterpreted Functions and Constants

Constants are function taking no parameters, so everything are functions.

Functions in first-order logic should be pure and total.

Use declare-sort to declare a unspecified type.

Arithmetic

Z3 supports integers and real numbers.

Nonlinear Arithmetic

A formula with expression * m n where m and n are neither numbers is called nonlinear. Nonlinear integer arithmetic is undecidable.

Z3 can only solve some of the nonlinear arithmetic problems.

Division

Z3 support real number division /, integer division div, modulo mod and remainder rem.

The result of division by zero is not specified due to all functions need to be total. It can be any result if Z3 wants. You may need a wrapper to avoid this.

第六周

Combination Theories

Combined Theory

For theory $T_1, T_2$, if $T_1\cap T_2 = \{=\}$, then the combined theory $T_1\cup T_2$ has signature $\Sigma_1\cup\Sigma_2$ and axioms $A_1\cup A_2$.

Nelson-Oppen Combination Method

$T_1\cup T_2$ is decidable if both $T_1$ and $T_2$ are quantifier-free, conjunctive fragments which are decidable and stably-infinite (see below).

Stably-Infinite Theories

A theory $T$ with signature $\Sigma$ is stably infinite if for every quantifier-free $T$-satisfiable $\Sigma$-formula $F$, there is a satisfiable $T$-interpretation with domain of infinite cardinality.

$T_E$, $T_A$ and $T_Z$ are stably infinite.

Nelson-Oppen Procedure

This procedure takes a formula $F$ in combined theory $T_1\cup T_2$, and returns an equisatisfiable formula $F_1\land F_2$ where $F_1$ is a $T_1$-formula and $F_2$ is a $T_2$-formula.

In short, the procedure can separate a formula in a combined theory into two formulas in each theory.

Step 1. Variable Abstraction

Procedure

In this step, our goal is to make every literal in $F$ belongs to either $T_1$ or $T_2$.

A function or a predict is a literal, so:

  • If function $f(\dots, t, \dots)$ is in $F$, where $f$ in $\Sigma_1$ but $t$ in $\Sigma_2$, then we introduce a new variable $w$ and replace $t$ with w, then conjoin $t = w$.
  • If predicate $p(\dots, t, \dots)$ is in $F$, where $p$ in $\Sigma_1$ but $t$ in $\Sigma_2$, we introduce a new variable $w$ and replace $t$ with w, then conjoin $t = w$.

Note that a variable is a logic symbol. When finished, every literal in $F$ belows to either $T_1$ or $T_2$. Note the $=$ cannot be viewed as a predicate here.

    • $F_2\land \alpha(V, E)$ is $T_2$-satisfiable.

Example

(Follows the example in step 1.)

  1. $F_1$ (which in $T_E$) is $f(x)\ne f(m)\land f(x)\ne f(n)$, and $F_2$ (which in $T_\mathbb Z$) is $1\le x \land x \le 2 \land m = 1\land n = 2$.
  2. Shared variables are $V = \{x, m, n\}$.
  3. For every possible partition on $V$, we can construct an equivalence relation. There are five possibilities.
    • $\{\{x\}, \{m\}, \{n\}\}$: No, because $F_2\land x\ne m \land x \ne n$ is not $T_\mathbb Z$-satisfiable. (Some items are omitted.)
    • $\{\{x\}, \{m =n\}\}$: No, because $F_2\land m=n$ is not $T_\mathbb Z$-satisfiable.
    • $\{\{x, m\}, \{n\}\}$: No, because $F_1\land x = m$ is not $T_E$-satisfiable.
    • Same for the remaining two possibilities.
  4. As all possibilities cannot ensure $F_1\land \alpha(V, E)$ is $T_E$-satisfiable and $F_2 \land \alpha(V, E)$ is $T_\mathbb Z$-satisfiable, the original formula $F$ is not $T_E\cup T_\mathbb Z$-satisfiable.

Efficiency Issue

The method above is impractical as we need to construct too many partitions.
