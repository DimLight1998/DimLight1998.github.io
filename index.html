Well-Founded Relations

A binary relation $\prec$ on $R$ is well-founded if for any subset of $R$, there is a "minimal" element, and any "decreasing" sequence from a certain element is finite. (We call $a$ is less than $b$ if $a \prec b$.)

Examples

  • less-than relation on $\mathbb N$ is well-founded, because for any element $n$ in $\mathbb N$, the decreasing sequence $n, n - 1, n - 2, \dots, 0$ is finite.
  • greater-than relation on $\mathbb N$ is not well-founded, because for any element $n$ in $\mathbb N$, there exists an infinite decreasing sequence $n, n + 1, n + 2, \dots$.
  • less-than relation on $\mathbb R^+$ is not well-founded.

Motivation

Given a program composed of many procedures with preconditions and postconditions annotations, how to prove the correctness of the program automatically?

VC Generation: Predicate Transformer

Given an assertion $Q$ and a program $c$, a predicate transformer is a function that produces another assertion.

Given an assertion $Q$ and a program $c$, we can construct a weakest precondition predicate transformer $wp(c, Q)$, such that:

  • $[wp(c, Q)]c[Q]$ is valid
  • for all $P$ such that $[P]c[Q]$ is valid, we have $P\Rightarrow wp(c, Q)$ (hence $wp(c, Q)$ is the weakest).

Similarly we have weakest liberal precondition $wlp(c, Q)$, and it doesn't require termination.

Similarly we have strongest postcondition $sp(c, Q)$.

Imp Language Array Extension

Add array read $x[a]$ and array write $x[a_1]:=a_2$ to the Imp language. Array writing operation will return an updated new array, and the original array was not modified.

第九周

Program Specification

The correctness of programs can be ensured by preconditions and postconditions, which are called program specifications.

Program specifications should be:

  • Declarative
  • Informative
  • Refutable

Notions

  • Environment: valuation of all variables in the program $\sigma$
  • Configuration: $\langle c, \sigma\rangle$, where $c$ is a statement
  • Small-step semantics: $\langle c, \sigma\rangle\rightarrow \langle c', \sigma' \rangle$
  • Big-step semantics: $\displaystyle{\langle c, \sigma \rangle \Downarrow \sigma'}$

Partial Correctness

Hoare Triples

A Hoare triple is in the form of $\displaystyle{\{P\}c\{Q\}}$, where:

  • $c$ is a program
  • $\{P\}$ are the preconditions
  • $\{Q\}$ are the postconditions
  • assertions in $\{P\}$ and $\{Q\}$ are in a first-order theory

Meaning of Hoare Triples

$\displaystyle{\{P\}c \{Q\}}$ is a predicate. If a program starts from an environment satisfying $P$, and if $c$ terminates, then the program will end in an environment satisfying $Q$.

However, we don't know what will happen if the program starts from an environment not satisfying $P$ or $c$ doesn't terminate.

Total Correctness

Total correctness id denoted as $[P]c[Q]$. It is stronger than partial correctness.

$[P]c[Q]$ is a predicate. If a program starts from an environment satisfying $P$, then $c$ terminates, and the program will end in an environment satisfying $Q$.

Church-Rosser 定理

对于一个表达式来说，只要它的求值是终止的，那么不管采用何种求值策略，最后总是会得到相同的值。

显然这个定理要在计算没有副作用时才能成立。

Scala 中的状态

Scala 中可以使用 var 关键字定义变量。变量可以随后被修改。这样就引入了可变的状态。

相等性

在引入可变性之前两个对象的相等是很好判断的。只要值相等，二者相等。这叫做 referential transparency。

引入了可变性之后两个对象是否相等要通过 operational equivalence 判断，依据是 是否存在能够将二者区分开来的操作序列。如果不存在这样的序列，就认为二者确实相等。

实际上证明两个对象相等是很麻烦的。

For Expression 的处理

首先可以注意到 map、filter 以及 flatMap 都可以由 for expression 加以实现。

Scala 编译器则会反过来将代码中的 for expression 翻译回 map、filter 和 flatMap：

  • for (x <- e1) yield e2 会被翻译为 e1.map(x => e2)；
  • for (x <- e1 if f; s) yield e2 会被翻译为 for (x <- e1.withFilter(x => f); s) yield e2，withFilter 是 filter 的 lazy 版本，如果 filter 后面要跟 map 或者 flatMap 操作的话这样做可以加速；
  • for (x <- e1; y <- e2; s) yield e3 会被翻译成 e1.flatMap(x => for (y <- e2; s) yield e3)。

根据上面的规则，可以把任意复杂的 for expression 分解为几个基础高阶函数操作。

另外，只要自己定义的结构具有 map、withFilter 和 flatMap 方法，就可以使用 for expression。

Scala 是纯 OOP 的吗

值类型可以被 OOP 实现

纯 OOP 语言指的是这种语言中所有的东西都是基于类和对象的。Java 不是纯 OOP 语言，因为其中的 primitive types 不是基于类或者对象的。

Scala 是纯 OOP 的，尽管它也有像 Java 中的 primitive types，但是 逻辑上 这些类型都是用 OOP 实现的。在编译时，编译器会特殊地对待这些类型，将其转化为更加底层的类型以保证运行效率。

OOP 可以实现诸如 boolean 和 int 这样的类型，并且实现一个原生类型时不需要依赖这个类型，即值类型可以被 OOP 体系实现。比如，可以用 OOP 实现皮亚诺算术进而实现整数，尽管这么做效率很低，但是确实是可行的。

函数类型可以被 OOP 实现

Variance 进阶

Scala 泛型类的协变逆变约束

Scala 中的数组是逆变的，但是不可变的列表（List）呢（答案见后）？大体上讲，所有的 mutable 类型都不应该被设计成协变的，而 immutable 类型只有在满足某些条件的情况下可以设计为协变的。

对于类型 A 和 B 有 A <: B，如果有某个类型构造子 C，则下面三种情况必有一个成立：

  • covariant：C[A] <: C[B]
  • contravariant：C[A] >: C[B]
  • nonvariant：上面两个都不成立

Scala 中，声明泛型类的时候可以声明这个泛型类对于类型参数是协变的、逆变的还是不变的：

  • convariant：class C[+A] { ... }
  • contravariant：class C[-A] { ... }
  • nonvariant：class C[A] { ... }
