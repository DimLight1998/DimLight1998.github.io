抽象最小不动点的计算

如果完全偏序集 $A$ 上没有无穷递增链，$f'$ 是 $A$ 上的单调函数，$a_0$ 是 $f'$ 的前不动点，那么迭代 $a_{i + 1} = f'(a_i)$ 即可在有限步骤内收敛到 $\text{lfp}_{a_0}f'$。

当 $A$ 上有无穷递增链的时候会比较麻烦，为此引入偏序集 $(A, \sqsubseteq)$ 上的 加宽算子 $\triangledown: A\times A\rightarrow A$，它应当满足：

  • $\forall x, y\in A, x\sqsubseteq x\triangledown y\land y\sqsubseteq x\triangledown y$
  • 对于任意的链 $(x_i)_{i\in \mathbb N}$，按照如下规则构造出来的链 $(y_i)_{i\in \mathbb N}$ 应当是递增的并且 $\exists k\in \mathbb N, y_{k + 1} = y_k$：
    • $y_0 = x_0$
    • $y_{i + 1} = y_i\triangledown x_i$

现在，还是给定完全格 $A$ 和上面的单调函数 $f':A\rightarrow A$，那么下面的递增链：

  • $y_0 = \bot'$
  • $y_{i + 1}=\begin{cases}y_i &\text{if }f'(y_i)\sqsubseteq y_i \\ y_i\triangledown f'(y_i)&\text{otherwise}\end{cases}$

将会在有限步骤内收敛到某个 $y_k$，这个 $y_k$ 是 $\text{lfp}f'$ 的上近似（$\text{lfp}f'\sqsubseteq y_k$）（实际上它是 $f'$ 的后不动点）。可以看到这个算法只是求出了不动点的近似。

形式化验证课程笔记 12

发表于 2019-05-13

第十二周

抽象解释

抽象解释 是对语义进行抽象分析的过程。它会将程序的解释从具体的空间映射到一个抽象空间中，可以用来搞静态分析。

数学基础

偏序

偏序关系是自反、传递、反对称的关系。

偏序集记作 $\displaystyle{(D, \sqsubseteq)}$。

格

如果一个偏序集满足：

  • 任意两个元素 $\displaystyle{a, b}$ 都可以导出一个最小上界 join：$\displaystyle{a\sqcup b}$
  • 任意两个元素 $\displaystyle{a, b}$ 都可以导出一个最大下界 meet：$\displaystyle{a\sqcap b}$

这样的偏序集，以及对应的 join 和 meet 定义构成了一个 格。

是偏序但是不是格的例子，其中 a 和 b 找不到 join 和 meet

完全格

对于一个格，如果它的所有元素构成的集合的 每个子集都有自己的最小上界和最大下界，则称为 完全格。完全格具有最小元 $\displaystyle{\bot = \sqcup\varnothing}$ 和最大元 $\displaystyle{\top = \sqcup D}$，其中 $\displaystyle{\sqcup X}$ 表示 $\displaystyle{X}$ 的最小上界，$\displaystyle{D}$ 是格的所有元素构成的集合。

链

链 是偏序集的子集，其中任意两个元素应当可以进行比较。

完全偏序

如果偏序集满足：

  • 具有最小元 $\displaystyle{\bot_D}$
  • 任意一个链 $\displaystyle{S}$ 都具有最小上界 $\displaystyle{\sqcup S}$（这个最小上界不一定要在 $\displaystyle{S}$ 中，但我不确定）

则这个偏序集为 完全偏序。

偏序但不是完全偏序的例子是具有自然序的自然数。因为存在无穷长的链，这样的链中不存在最小上界。不过可以通过添加 $\displaystyle{\infty}$ 元素来让集合变成是完全偏序的。

函数

单调性

对于偏序集 $\displaystyle{(D, \sqsubseteq)}$ 和 $\displaystyle{(D', \sqsubseteq')}$，若函数 $\displaystyle{f: D\mapsto D'}$ 满足 $\displaystyle{x\sqsubseteq y \rightarrow f(x)\sqsubseteq'f(y)}$，称其为 单调的。

连续性

对于完全偏序的偏序集 $\displaystyle{(D, \sqsubseteq)}$ 和 $\displaystyle{(D', \sqsubseteq')}$，若函数 $\displaystyle{f: D\mapsto D'}$ 满足：对于 $\displaystyle{D}$ 中的每条链 $\displaystyle{S}$，$\displaystyle{f}$ 操作和 $\displaystyle{\sqcup}$ 操作是可交换的（即 $\displaystyle{\sqcup f(S)}$ 存在且 $\displaystyle{\sqcup f(S) = f(\sqcup S)}$），则称其为 $\displaystyle{\sqcup}$-连续的。

单调但是不连续的例子：$\displaystyle{f:\mathbf N\cup \{\infty\}\mapsto \{0, 1\}}$，$\displaystyle{f}$ 将 $\displaystyle{\infty}$ 映射为 $\displaystyle{1}$，其它映射为 $\displaystyle{0}$。

单调同时连续的例子：修改上一个例子，将小于某个值的所有数映射为 $\displaystyle{0}$，其它映射为 $\displaystyle{1}$。

  • 简单的类型系统表达能力不够，为了让类型之间能够“兼容”，需要引入类型之间的 subtyping 关系，让一种类型能够“替代”另一种类型出现。Subtyping 关系构成类型上的偏序关系。
  • 引入 subtyping 后类型系统依然安全，表达能力没有增强，可以通过 coercion semantics 将带有 subtyping 关系的语言翻译到简单类型语言。

  • 使用 lambda 演算可以对 OOP 进行编码来证明其性质，但是研究 OOP 的类型系统时通常会搞个新的演算系统。

TaPL 读书笔记 4 Subtyping

发表于 2019-05-08 | 更新于 2019-05-28

笔记对应原书 15-19 章。

上一部分（Simple Types）的重点回顾：

  • 使用 类型系统 可以 静态地 检查出一部分可能会出错的程序，而我们希望运行过程中会 stuck 的程序能够被拒绝。
  • 语言的定义会包含一系列的 typing rules，根据这些 typing rules 能够推断出一些 terms 的类型，这些 terms 是 well typed terms。
  • 类型系统依据 progress 性质 和 preservation 性质 来保证程序不可能 stuck，但不一定保证程序一定会终止。合格的类型系统必须保证这两个规则成立：
    • Progress：一个 well typed term 要么是 value，要么就能继续求值。
    • Preservation：一个 well typed term 如果求值了，那么得到的 term 还是 well typed 的。
  • 如果每个 well typed term 的类型唯一，则类型系统有 uniqueness 性质。
  • 如果每个 well typed term 求值必定终止，则类型系统有 normalizable 性质。
  • 我们可以给无类型 lambda 演算加入 函数类型，但是这样的演算系统没有 well typed terms。
  • 我们可以给带有一些简单类型的 lambda 演算加入各种扩展，实现 let binding、tuple、record、variants、语言内建递归等。
  • 在保证类型安全的情况下，我们可以给简单类型 lambda 演算加入 引用 和 异常 的扩展。

在这一部分的笔记开始前插一句，类型系统分为 nominal 和 structural 两种，差异是类型相同时如何比较。可以定义两个除了名字不同，其它都一样的类型，如果类型系统认为两个类型是同一个类型，这种类型系统就是 structural 的（比如 Haskell），否则就是 nominal 的（比如 C++、Java）。

我发现用多了多重标题后，一篇博客会充斥着无聊的内容，所以这次我尝试不写多重标题。

前面我们为无类型 lambda 演算引入了一个简单的类型系统，它确实实现了目的——拒绝掉会 stuck 的程序。然而这个系统在一些情况下用起来很麻烦，表现力不够强，如果一个函数要接受某种记录类型，那即使传入了多了几个域的“兼容”的记录类型也是不行的，然而这个程序明显是正确的。解决这个问题的办法是引入 subtyping，对于两个类型 S 和 T 用 S <: T 表示 S 是 T 的子类型，意思是 所有 T 可以出现的地方都可以用 S 进行替换，任何类型为 S 的 term 的类型也为 T，就像 OOP 里面的继承那样。显然 subtyping 关系是自反、传递、反对称的（构成偏序集）。

可以证明引入了 subtyping 后类型系统还是可以发挥作用，依然满足 progress 和 preservation 性质。另外，subtyping 并没有增强类型系统的功能，只是提高了它的表达能力。通过一种叫做 coercion semantics 的东西，可以对给定的 subtyping 规则，构造出一组将其翻译到不带 subtyping 的类型系统的语言的规则，例如将有 subtyping 的 lambda 演算翻译到简单类型 lambda 演算，所以它们的表达能力是一样的。

为了便于讨论一般还会引入两个类型 Top 和 Bot。其中所有类型都是 Top 的子类型（这也意味着如果你知道一个值的类型是 Top，相当于你什么都不知道）；Bot 是所有类型的子类型（它没有 closed form term，一般用于表示不会返回的计算过程，例如不会终止的过程或者一定会抛出异常之类的）。在一些 OOP 语言中所有的类都有个父类 Object，这就相当于 Top。

另外还可以引入 intersection type 和 union type 的概念。由于类型可以看作是 term 的集合，subtyping 关系可以构造偏序集，因此可以定义两个类型的 meet 是它们的 intersection type，包括了同时属于这两个类型的 term（即集合的交）；两个类型的 join 是它们的 union type，是 terms 的集合的并。Union type 和 variant 的区别在于前者不需要 tag（当然也就没法区分到底是哪个类型）。关于 intersection type 有一些定理：

TaPL 读书笔记 3 Simple Types

发表于 2019-05-08 | 更新于 2019-05-24

笔记对应原书 8-14 章。

上一部分（Untyped Systems）的重点回顾：

  • term 是具有实际意义的合法程序单元
  • 语法 是对 term 的定义；定义一般是递归的，方便用归纳法证明相关性质
  • 语义 给出了对 term 的求值规则，小步操作语义给出了 term 之间的变换规则
  • value 是 term 的一个子集，value 不能继续求值
  • 如果找不到对 term 继续求值的规则，这个 term 是 normal form，但不一定是 value（如果确实不是 value，称为 stuck）
  • 即使找得到对 term 继续求值的规则，也有可能出现求值 无法终止 的情况
  • 书中使用的 lambda 演算是 call by value 的
  • 无类型 lambda 演算可以用来编码各种东西，也能表示递归；也可对其进行语法和语义的扩展

为什么需要类型

类型系统是干嘛的

回到最开始对类型系统的定义：

类型系统本质上是一个轻量级的形式化方法，用来拒绝一部分可能存在错误的程序。它不会引入很多复杂性，因此是轻量级的。具体要拒绝具有哪些性质的程序由语言的定义者定夺取舍。

在无类型 lambda 演算中，会出现以下两种情况：

  1. 存在不是 value 的 norm form
  2. 存在不能终止求值的 term

当我们希望程序总是终止并且能够求出一个 value 时，我们不希望这两种情况出现。写了一个程序之后，我们想让某个东西告诉我们这个程序是否会 stuck。类型系统可以实现这一点。通过引入类型系统，我们能够“静态地”在实际运行程序前，就分析出这个程序是否可能存在上述两种情况，据此决定是否要拒绝它。

简单的类型系统可以让以上两点被拒绝，如果类型系统更加复杂的话，第二点可能无法被拒绝。

类型的一些性质

在导出类型系统的性质之前有必要引入相应的符号和定义：

  • term $t$ 具有类型 $T$ 记作 $t:T$
  • 如果存在类型 $T$ 使得 $t:T$，则称 term $t$ 是 well typed 的

确实可以构造出符合语法规定的，但是不是 well typed 的 term，比如 succ(true)，这种 term 在求值时会 stuck。

在有类型系统的语言的相关性质的证明中，一个常用的套路是根据类型进行归纳。例如我们要证明 term $t$ 具有某个性质，首先分析 $t$ 可能的类型有哪些，然后对于每种类型，分析出它为什么会具有这种类型（通过哪一条 typing rule），完成这一步之后，$t$ 的形式就固定了（比如 $t$ 必然是 $\texttt{if }t_1\texttt{ then }t_2\texttt{ else }t_3$ 结构的），然后归纳证明即可。

类型系统的一个性质是 每个 well typed term 只可能有一种类型，并且只有一棵推导出该类型的推导树。不过并非所有的类型系统都会满足这个性质（比如包括子类型的语言就不会）。这个性质称为 uniqueness。具有 uniqueness 性质的类型系统可以让编译器只扫描一遍 AST 就分析出各个 term 的类型，而不会有“猜测”的过程。

类型系统的另一个性质是保证了每种类型对应的 value 一定会具有某种形式，就像 inversion lemma 给出的那样。这个性质称为 canonical forms。

语言的语义

程序的语义指的是 term 如何被求值。程序的运行过程实际上是对 term 的变换过程，最终的目的是将 term 变换到一个 value。所有 value 是所有 term 构成集合的一个子集，该子集是人工定义出的。程序的语义提供了 term 转换为 value 的过程中每一步的依据。

语义的描述方式有以下几种（我只看懂了操作语义）：

  • 操作语义：操作语义给出了一个状态机以及状态变迁的规则，程序的执行即为状态的变迁。对于简单的语言来说，term 就可以作为状态。
  • 指称语义：指称语义对 term 的解释是将其映射到一个 semantic domain 上，这个 domain 中的东西都是一些数学对象。
  • 公理语义：这个就更加抽象了，将规则本身作为语言的定义，将 term 的性质作为 term 的定义。

操作语义又分为 小步语义 和 大步语义：小步语义给出了 term 计算到 value 过程中每一步的规则，而大步语义直接给出了怎么到最后的结果。

书中用到的所有语义都是操作语义。

类型系统

什么是类型系统

类型系统本质上是一个轻量级的 形式化方法，用来拒绝一部分可能存在错误的程序。它不会引入很多复杂性，因此是轻量级的。具体要拒绝具有哪些性质的程序由语言的定义者定夺取舍。

没有被拒绝的程序一定不会发生某些特定的错误，例如，如果类型系统要求加号两边都是数字，那么程序运行时不可能出现一个数字要加上字符串导致的错误，这种错误在编译阶段就被搞掉了。

使用类型系统的好处

  • 检测潜在的错误。
  • 提高抽象度：使用类型系统会强迫程序员写出更加模块化的程序。
  • 作为永不过时的文档。
  • 保障语言安全：安全有各种定义。一个定义方式是程序产生错误的时候（比如数组越界）能够被检测到（Java 会抛异常，因此是安全的；C++ 会继续执行但是后果不确定，因此是危险的；两种分别叫做 trapped error 和 untrapped error）。使用类型系统可以在一定程度上提高语言的安全性，比如在提供了数组抽象的语言中都是通过语言的接口来“安全地”对底层的内存进行操作、提供了函数抽象的语言会帮程序员管理栈帧，而像汇编这种语言就要手动控制内存进行数组操作、函数的调用栈的控制，不安全。
  • 提高运行效率：例如使用静态类型系统能够在运行时不再操心对象的类型 tag，减少运行时的开销。

前置数学知识

集合与关系

关系是集合的笛卡尔积的子集。

集合上的谓词可以看作是一元关系 $P$，对于属于关系 $P$ 的元素，谓词返回真，否则返回假。

很多东西都可以用关系来建模，例如 $\Gamma\vdash s: T$ 就可以看作是某个三元关系中的元组。

函数实际上是一个二元关系 $R: S\rightarrow T$，函数可以是 partial 的（定义域是 $S$ 的真子集）或者 total 的（定义域就是 $S$）。可以使用 $f(x)\uparrow$ 表示函数在 $x$ 上没有定义，用 $f(x)\downarrow$ 表示有定义。

序

关系可以有 reflexive，symmetric 和 transitive 性质，比较简单。

Antisymmetric 说的是两个不同元素构成的二元组不会都满足 $R$：

  • 如果一个关系是 reflexive 和 transitive 的，这个关系就是 pre-order 的；
  • 如果这个关系还是 antisymmetric 的，这个关系就是 partial order 的；
  • 如果任意两个元素都能进行比较，这个关系就是 total order 的。

可以使用 $\le$ 和 $\lt$ 来描述 pre-order 关系中的元素，区别是元素能否相同。

数学归纳法

数学归纳法不一定是对自然数进行归纳，也可以对由自然数形成的元组进行归纳。对于 超限归纳法（Complete Induction） 这也是成立的。

Well-Founded Relations

A binary relation $\prec$ on $R$ is well-founded if for any subset of $R$, there is a "minimal" element, and any "decreasing" sequence from a certain element is finite. (We call $a$ is less than $b$ if $a \prec b$.)

Examples

  • less-than relation on $\mathbb N$ is well-founded, because for any element $n$ in $\mathbb N$, the decreasing sequence $n, n - 1, n - 2, \dots, 0$ is finite.
  • greater-than relation on $\mathbb N$ is not well-founded, because for any element $n$ in $\mathbb N$, there exists an infinite decreasing sequence $n, n + 1, n + 2, \dots$.

Motivation

Given a program composed of many procedures with preconditions and postconditions annotations, how to prove the correctness of the program automatically?

VC Generation: Predicate Transformer

Given an assertion $Q$ and a program $c$, a predicate transformer is a function that produces another assertion.

Given an assertion $Q$ and a program $c$, we can construct a weakest precondition predicate transformer $wp(c, Q)$, such that:

  • $[wp(c, Q)]c[Q]$ is valid
  • for all $P$ such that $[P]c[Q]$ is valid, we have $P\Rightarrow wp(c, Q)$ (hence $wp(c, Q)$ is the weakest).

Similarly we have weakest liberal precondition $wlp(c, Q)$, and it doesn't require termination.

Similarly we have strongest postcondition $sp(c, Q)$.

第九周

Program Specification

The correctness of programs can be ensured by preconditions and postconditions, which are called program specifications.

Program specifications should be:

  • Declarative
  • Informative
  • Refutable

Notions

  • Environment: valuation of all variables in the program $\sigma$
  • Configuration: $\langle c, \sigma\rangle$, where $c$ is a statement
  • Small-step semantics: $\langle c, \sigma\rangle\rightarrow \langle c', \sigma' \rangle$
  • Big-step semantics: $\displaystyle{\langle c, \sigma \rangle \Downarrow \sigma'}$

Partial Correctness

Hoare Triples

A Hoare triple is in the form of $\displaystyle{\{P\}c\{Q\}}$, where:

  • $c$ is a program
  • $\{P\}$ are the preconditions
  • $\{Q\}$ are the postconditions
  • assertions in $\{P\}$ and $\{Q\}$ are in a first-order theory

Meaning of Hoare Triples

$\displaystyle{\{P\}c \{Q\}}$ is a predicate. If a program starts from an environment satisfying $P$, and if $c$ terminates, then the program will end in an environment satisfying $Q$.

However, we don't know what will happen if the program starts from an environment not satisfying $P$ or $c$ doesn't terminate.

Total Correctness

Total correctness id denoted as $[P]c[Q]$. It is stronger than partial correctness.

$[P]c[Q]$ is a predicate. If a program starts from an environment satisfying $P$, then $c$ terminates, and the program will end in an environment satisfying $Q$.
