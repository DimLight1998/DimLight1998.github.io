SQL 语言与数据库完整性和安全性

数据库是概念，SQL 是实现。二者不是同一个抽象层上的东西，看的时候注意这一点。

数据库完整性概念及完整性约束规则

数据库完整性

DBMS 应当保证数据库始终具有某种性质，这种性质一般和正确性、有效性和一致性有关。

完整性涉及到语义上的完整性和数据库实现上的完整性。

为什么会引发完整性问题

语义相关：错误数据输入、操作失误

实现相关：程序处理失误

DBMS 应当防止这些错误导致的语义不合理现象。

如何保证完整性

DBA 会通过 DDL 定义语义完整性。用户修改数据时，DBMS 仅在通过检查后允许修改。

完整性约束条件是四元组 $(O, P, A, R)$，分量含义分别为数据集合（约束对象）、谓词条件（什么样的约束）、触发条件（何时进行检查）和响应动作（不满足时触发的动作）。

数据库完整性分类

按约束对象分类

  • 域完整性：施加于某一列的约束。只给了一列数据也能判定某记录是否符合条件。例如要求某列的值必须大于 10。
  • 关系完整性：施加于记录上的语速。例如要求某记录的某两个域的和必须大于 10。

按约束来源分类

  • 结构约束：来自模型的约束，例如主键不为空、外键必须是候选键等。
  • 内容约束：用户定义的语义约束。

按约束状态

  • 静态约束：数据在任意时刻都应该满足的约束。
  • 动态约束：数据在进行修改时应当满足的约束。例如约束某一列的值只能升不能降。

SQL 语言之复杂查询与视图

In 子查询

子查询

子查询存在是为了应对查询中和集合有关的查询，例如元素是否属于某集合、集合是否为空等。

子查询返回的是一个集合。

In 子查询

判断一个集合是否在给定子查询中：$$Expression [not] in $$SubQuery。一般在 where 子句中出现。

在有子查询的情况下，查询可以分为外层查询和内层查询。如果内层查询不依赖外层查询（不使用外层查询的信息），称为 非相关子查询，否则是相关的。相关子查询中，信息只能从外层向内层传递，不能反过来。

Theta Some/All 子查询

这个查询主要判断集合之间的包含关系。假设 θ 表示一个二元比较关系，例如等于、大于、小于等于等。那么有两种 theta 查询：$$Expression θ some $$SubQuery 和 $$Expression θ all $$SubQuery。前者在子查询中某个值满足 θ 关系时就能成立，后者需要在子查询的所有值都满足关系的时候才成立。一般在 where 子句中出现。

  • $$Expression in $$SubQuery 可以看做是 $$Expression = some $$SubQuery。
  • $$Expression not in $$SubQuery 可以看作是 $$Expression <> all $$SubQuery。

关系模型之关系代数

关系代数都是对集合进行操作的：输入和输出都是集合，从已有集合得到想要的集合。

下面的操作按照难度递增，分别是基本操作、扩展操作和复杂扩展操作。

什么是关系代数

关系代数通常以一个或多个关系作为输入，结果是新的关系。关系代数是一种抽象的语言。

关系代数可以分成 集合操作（并交差积） 和 纯关系操作（投影、选择、连接、除）。

DBMS 可以先解析 SQL 语言，将其翻译为多个关系代数操作的序列。这样，只要实现了基本的一些关系代数操作就可以实现一个 DBMS 了。

并相容性的概念

两个关系是并相容的，如果它们的属性数目相同，相同位置上的属性的类型也相同。

简单来说，两个关系并相容，表示可以把一个关系纵向拼接到另一个关系下面形成更大的关系。

并相容性是其它很多操作的基础。

关系运算分为关系代数（见上次的笔记）与关系演算。关系演算是以谓词演算为基础的，按照谓词变量不同分为 元组演算 和 域演算。从名字可以看出一个是针对行，一个是针对列。

后面可以看到在一定的条件下，关系代数和关系演算是可以相互转化的。

元组演算也是用来找出某个关系中的记录的。这里的“元组”就是记录的意思，但是名字听起来让人不爽。

关系模型之基本概念

什么是关系模型

关系模型的提出

提出：关系模型从对表的处理中提出，多数商业数据库在使用基于关系模型的数据库。

关系与关系模型

简单来说一张表就是一个关系。

关系模型就是处理表的模型，包括三个部分：

  • 结构：描述数据库、数据的结构
  • 操作：描述表之间可能进行的操作（运算）
  • 完整性约束：描述上述操作需要遵循的条件

关系运算

关系运算分为 关系代数 和 关系演算，关系演算又分为 元组演算 和 域演算（一个按行一个按列）。这三种演算都有对应的语言实现。

  • 关系代数 一次对一个集合进行操作（很多非关系的代数一次对单条记录进行操作）。
  • 元组演算 是基于逻辑的演算，域演算比较少见。

数据库系统的结构抽象与演变

数据库系统的标准结构

三层次抽象

数据的存储可以看做有三个抽象层次：

  • 外部层次（用户层次）：DBAP 可以看到的层次，一般是所有数据的一部分。
  • 概念层次（逻辑层次）：DBMS 使用到的层次，包括对应的约束和关联。
  • 内部层次（物理层次）：数据在介质上的存储，包括路径、索引等。

模式与视图

  • 模式（Schema）：对数据库中的数据的结构化描述。
  • 视图（View）（又称数据）：通过某种表现形式表现出的数据库中的数据。它是一种表现。

将三层次抽象和模式视图结合，排列组合一下可以得到六个概念：内部模式、内部视图、概念模式、概念视图、外部模式、外部视图。当然知道这几个概念并没有什么用。

在不指明抽象层次的情况下，模式指的是概念模式，视图指的是外部视图。
