数据库索引技术

索引的概念和作用

什么是索引及其特点

索引用来在不用遍历所有记录的情况下快速定位所需记录在整个数据库中的位置。它是一种辅助结构，存在与否不改变数据表本身的组织。

索引由一系列 索引项 构成，每个索引项包括 索引字段（记录的被索引字段的值）和 行指针（指向记录的位置）。

存储索引项的文件称为 索引文件，数据库本身称为 主文件。

特点：

  • 一张表上可以对多个不同属性（及其集合）建立多个不同的索引文件。
  • 索引文件比表要小很多，一般可以全部装入内存。
  • 对主文件的更新应当同步更新索引文件。

索引文件的组织

索引文件有 排序索引 和 散列索引 两种组织方式，依据都是其中的索引字段。

对索引文件衡量标准

  • 时间：访问、插入、删除、更新索引文件对应的时间。
  • 空间：引入索引后增加的空间。
  • 便利程度：能否支持单一限制的查询；能否支持范围查询。

关于索引的一些概念

  • 排序码：主文件进行排序存储时用到的属性组
  • 索引码：索引字段，不一定是唯一的
  • 搜索码：用于在主文件中查找记录时用到的属性组

SQL 中索引的创建和使用

SQL 在用户定义主键后会自动创建 主索引。用户也可以手动创建索引或者撤销。SQL 会自动维护索引文件和主文件的一致性。

用 drop index 即可撤销索引。

稀疏索引与稠密索引

如果索引文件中包括了被索引字段的所有值，这样的索引策略称为 稠密索引，反之称为 稀疏索引。注意稠密索引的索引项数量不一定等于表中记录的数量，因为索引码可能有重复。

当使用 稀疏索引 时，为了找到需要的记录，应当先使用稀疏索引找到比所需记索引码小的最大的记录，然后往后顺序查找。

当使用 稠密索引 时，为了找到需要的记录只要看一下稠密索引中是否有对应的记录的索引码即可。没有的话就表示表中没有。

主文件在索引码上的排序

在使用稠密索引时，如果表中有被索引的字段相同的记录：

  • 如果索引项不允许重复，那么可以考虑将主文件按照索引码进行排序，索引项指向排序后与之相等的第一条记录。
  • 如果索引项允许重复，则主文件不用进行排序。可以建立和记录数目相同的索引项。
  • 如果索引项不允许重复，主文件不能进行排序，那么要引入一个中间层：每个索引项指向一个链表，链表中的每个元素再次指向一条记录。

主索引与辅助索引

主文件是按照存储块进行存储的。前面说到的索引都是对每条记录进行索引，但是这不是必须的。实际上可以只对主文件占用的每个存储块进行索引，每个索引项对应一个索引块，指针指向索引块的起始位置对应的记录（称为 锚记录）。

这种策略称为 主索引，它是一种稀疏索引。主索引使用的索引码一般是表的排序码，很多时候还是主码。主索引只能有一个。

辅助索引 是索引码不是排序码的索引，它是稠密索引。有时候使用辅助索引的效率会很高。尽管实现上可以将辅助索引指向块，但是一般实现为指向记录，这样更快。

聚簇索引与倒排索引

聚簇索引与非聚簇索引

如果索引中临近的记录在表中也是临近的，这样的索引称为 聚簇索引，否则为 非聚簇索引。

当使用聚簇索引时索引码一般是排序码。一张表只能有一个聚簇索引。主索引是聚簇索引，辅助索引是非聚簇索引。

聚簇索引可以用来修改表中的数据，例如要插入一条记录的时候就是通过聚簇索引完成的。但是非聚簇索引只能用来查询，不能用来修改。

倒排索引

倒排索引是来自文档检索的概念，它希望的是优化给定关键字时查找对应文档的效率。

  • 正排索引：存储每个文档对应哪些关键字
  • 倒排索引：存储每个关键字对应哪些文档

数据库物理存储

存储体系回顾

数据库一般存放在磁盘上（也有内存数据库），然后将内存作为缓冲区。这样可以兼顾大容量和高速度两个方面。

磁盘管理：操作系统将磁盘分成若干个块。一个文件的存储可能涉及到多个块。块之间按照类似单链表的方式进行组织。第一个块的位置信息存储在目录文件中。

内存管理：操作系统将内存分成若干个页。内存中只维护了一部分页面，如果要使用的页面不在内存中就要使用到页面交换。

磁盘的结构与特性

扇区是物理概念，块是逻辑概念。一个块是由多个 连续的 扇区实现的。

物理上优化读写效率的一些考虑：

  • 降低 IO 次数。
  • 降低排队等待时间。
  • 降低寻道或旋转延迟时间，这可以通过将文件在连续的块上进行存储、在同一个柱面的不同磁道上进行存储、在多个磁盘上并行存储实现。

RAID 技术是用来处理多个磁盘并行存储的技术。主要是要考虑到 并行处理 和 可靠性 两个方面。

查询实现的基本思想

数据存储的映射关系

在文件系统中，OS 对于每个文件都维护了一个起始块的地址。在 DBMS 中，数据库也为每条记录记录了对应块的地址。如果记录已经在内存中被缓存起来，那么 DBMS 还会维护每条记录对应的内存地址。

数据存储与查询实现的基本框架

存储管理器管理磁盘；缓冲区管理器管理内存；内存作为磁盘的缓冲区。数据库将数据、数据格式和数据控制信息三者分开存储，使用索引/文件和记录管理器进行管理，并且分别由外界的 DML、DDL 和 DCL 进行控制。

记录与表在磁盘上的存储

逻辑上，数据是按照 表-记录-属性值 进行存储的；物理上，数据存储在磁盘块中。

记录在磁盘上的存储可以使用两种策略：

  • 定长记录：为每个记录统一规定长度。
  • 变长记录：每条记录的长度不固定，通过分隔符进行分隔，或者使用类似目录的东西存偏置。

根据一条记录是否能够跨越块进行存储可以分为 跨块存储 和 不跨块存储。在跨块存储时，需要使用指针连接各块。这更加节省磁盘空间，但是读取效率存在问题。

后面的笔记中 磁盘块 表示的就是 OS 管理文件的单位；数据块 是存放数据的磁盘块；索引块 是存放索引的磁盘块。

关于一张表占用的磁盘块的分配方法有：

  • 连续分配：分配连续的磁盘块以存储这张表（扩展上会有困难）。
  • 链接分配：每个块中存放指向下一个块的指针（访问速度上会有问题）。
  • 按簇分配：一簇是若干连续的磁盘块。簇之间使用指针连接。
  • 索引分配：索引块中存放指向实际数据块的指针。

模式分解存在什么问题

当关系模式不满足前面提到的若干范式的时候就应当进行分解。

模式分解存在的问题

什么是模式分解

一个关系模式可以分解为几个更小的关系模式。这几个更小的关系模式的属性集的并应当是原来的关系模式的属性集，并且彼此之间互相不包含。

给定了一个关系模式的分解 $\displaystyle{\rho}$。对于一个关系 R，将它按照方案分解后得到的多个子关系自然连接起来，会得到另一个关系 $\displaystyle{m_\rho(R)}$，这个关系和 R 不一定相等。它们之间的差异正是要研究的。

  • 如果 $\displaystyle{m_\rho(R)}$ 中多出来了一些错误的记录，这种分解称为是 有损的。
  • 如果 $\displaystyle{m_\rho(R)}$ 相对于 R 丢失了某些函数依赖，这种分解不是 保持依赖 的。

关于 $\displaystyle{m_\rho(R)}$ 的一些性质：

  • R 中的记录必然出出现在 $\displaystyle{m_\rho(R)}$ 中。
  • $\displaystyle{m_\rho(R) = m_\rho(m_\rho(R))}$
  • 将 $\displaystyle{m_\rho(R)}$ 投影到各个小关系模式上的结果和将 R 投影上去的结果一致。

无损连接分解及其检验算法

什么是无损连接分解

对于关系模式 R 的任意关系 r，以及给定的分解方案 $\displaystyle{\rho}$，如果 $\displaystyle{m_\rho(r) = r}$，则称分解是无损连接分解。

函数依赖及其公理/定理

什么是函数依赖

函数依赖的定义

对于具有多个属性的关系模式 R，如果对于属性集合中的两个子集 A 和 B 和任意的关系 r，r 中不可能出现两个记录使得 A 中的值相等而 B 中的值不等，则称 A 函数决定 B，B 函数依赖于 A，记作 $\displaystyle{A\rightarrow B}$。

另一种说法是，如果将 A 和 B 单独拿出来做一个关系，这个关系中 A 可以作为主键。

函数依赖的性质

  • 有些函数依赖关系中 $\displaystyle{B\subseteq A}$，这种称为是 平凡 的函数依赖，反之为不平凡的函数依赖。
  • 即使两个属性之间存在函数依赖关系，即 $\displaystyle{A\rightarrow B}$，这不意味着 A 和 B 不能在关系的多条记录中出现。
  • 两个属性子集可能相互函数依赖。
  • 如果关系中，A 上没有相同的两个记录出现，那么任意属性子集都函数依赖于 A。
  • 有基于 模式 的函数依赖，要求对所有可能的关系都成立；也有基于 关系 的函数依赖，只要求对这一个关系成立即可。

部分函数依赖与完全函数依赖

如果 $\displaystyle{A\rightarrow B}$，但是存在 A 的真子集 A' 使得 $\displaystyle{A'\rightarrow B}$，则称 B 部分函数依赖 于 A。如果不存在这样的子集，此时为 完全函数依赖。

传递函数依赖

函数依赖关系具有传递性：$\displaystyle{A\rightarrow B, B\rightarrow C \Rightarrow A\rightarrow C}$。

如果加上一些更强的限制：$A\rightarrow B$ 和 $B\rightarrow C$ 都是非平凡的，且 $C\nsubseteq A$，$B\nrightarrow A$，则称 C 传递函数依赖 于 A。

函数依赖相关的几个重要概念

把数据库中的相关概念用函数依赖理论重新定义了一下：

  • 候选键：能够 完全决定 关系模式中所有属性的属性组称为关系模式上的 候选键。
  • 主键：对于一个关系模式，可以选取其任意一个候选键作为 主键。
  • 主属性：包含于任意一个候选键中的属性称为 主属性，否则为 非主属性。
  • 超键：超键和候选键类似，但是移除了完全决定的限制。超键是某个候选键对应属性集的超集。
  • 外键：某个属性组不是当前关系模式的候选键，但是是另一个关系模式的候选键，称为当前关系模式的 外键。

其它的一些概念：

  • 逻辑蕴涵：设 F 是关系模式 R 上函数依赖关系的集合（不一定是完整的）。对于 R 的两个属性组 A 和 B，如果从 F 中能够推理出 $\displaystyle{A\rightarrow B}$，则称 F 逻辑蕴涵 $\displaystyle{A\rightarrow B}$，记作 $\displaystyle{F\vDash A\rightarrow B}$。
  • 闭包：设 F 是关系模式 R 上函数依赖关系的集合（不一定是完整的）。F 逻辑蕴涵的所有函数依赖关系构成的集合称为 F 的闭包。

可以看到，函数依赖是属性集子集上的一个关系，起码它是传递、自反的。

关系模式设计之规范形式

下面的范式都是针对 关系模式 而言的。就是说你可以通过一张表来判定这张表是否符合某个范式。

什么是第一范式

如果关系模式中关系的每个属性都是原子而不可分的，这种关系模式符合 第一范式。

第一范式不允许复合属性和多值属性。

不符合第一范式时的处理手段

第一种办法是转换为符合第一范式的关系模式。

  • 对于多值属性，可以将该属性和主键一起新建一个关系。
  • 对于复合属性，要么将其打包在一起得到一个属性，要么将其拆开形成多个属性。

第二种办法是使用其它数据模型进行处理。

什么是第二范式

如果关系模式是符合第一范式的，并且关系中每个 非主属性 都 完全依赖 于 候选键，则该关系模式符合 第二范式。

解释：候选键要求是最小的，能够决定其它所有属性的属性组。对于唯一确定关系中的一条完整的记录，候选键是必需的；但是唯一确定关系中某个属性对应的一条记录，候选键中则可能有多余的属性。

举个不满足第二范式的例子，假设关系模式是（学号，姓名，班级，课程，成绩）。根据这个关系中的函数依赖关系，可以将（学号，课程）作为一个候选键。因此姓名是非主属性，但是（姓名）这个属性是部分依赖于（学号，课程）的，实际上只要（学号）就能唯一确定（姓名）了。

不符合第二范式的关系中存在 非受控冗余。在上面那张表中，需要时刻维护一个学号只对应一个姓名，但是相同的（学号，姓名）二元组在关系中多次出现。

不符合第二范式时的处理手段

将这个关系分解为多个小的关系即可。

什么是第三范式

如果关系模式是符合第二范式的，并且关系中不存在传递依赖，则该关系模式符合 第三范式。

关于传递依赖部分，更加严谨的定义是关系模式中 不存在 这样的情况：X 是候选键，Y 是属性组，A 是非主属性，$\displaystyle{A\notin X, A\notin Y, Y\nsubseteq X, Y\nrightarrow X}$，使得 $\displaystyle{X\rightarrow Y, Y\rightarrow A}$ 成立。

一个不满足第三范式的例子：关系模式为（学生学号，系号，系主任）。它符合第二范式。此时（学生学号）是候选键，系主任是一个非主属性。这个关系模式中有传递依赖（学生序号）$\rightarrow$（系号），（系号）$\rightarrow$（系主任），因此不符合第三范式。

不符合第三范式也会导致出现非受控冗余。

不满足第三范式时的处理手段

将关系拆分成多个关系。最极端的方式是只要有个函数依赖就进行拆分，直到最后的关系中每个只有一个函数依赖关系。可以在此基础上进行合并。

和第二范式的比较

我感觉第二范式不允许某个属性函数依赖于候选键的真子集，第三范式不允许某个非主属性属性函数依赖于其它的非主属性（此处不严谨）。

什么是 Boyce-Codd 范式

如果关系是符合第一范式的，并且对于关系中的任何函数依赖 $\displaystyle{X\rightarrow Y}$，当 $\displaystyle{Y\nsubseteq X}$ 时必有 X 包含某个候选键，则称关系符合 Boyce-Codd 范式。

Boyce-Codd 范式比第三范式更加严格。如果一个关系不满足第三范式，则必不满足 Boyce—Codd 范式。

不符合 Boyce-Codd 范式的例子：对于关系模式（城市，街道，邮政编码）来说，候选键是（城市，街道），但是因为（邮政编码）可以唯一确定（城市），这是一个函数依赖，但是被依赖的属性组不包括候选键，因此不符合 Boyce-Codd 范式。

不满足 Boyce-Codd 范式时的处理手段

将不包含候选键的函数依赖单独拿出来组成一个关系，将包含候选键的函数依赖组成另一个关系。

数据库设计过程

数据库设计的过程：

  1. 需求分析
  2. 概念数据库 设计：得到概念模型，例如 E-R 图
  3. 逻辑数据库 设计：建立关系模型，得到三层抽象中的外模式
  4. 物理数据库 设计：依赖于具体的 DBMS 组织物理数据

如何做需求分析

需求分析最后要提交 需求分析报告。

如何做概念数据库设计

概念数据库设计最后要提交 概念数据库设计报告。应当清晰地包含：

  • 实体的发现、划分、定义
  • 实体属性的发现、划分、定义
  • 实体联系的发现、划分、定义
  • 外部视图和概念视图的定义

并且使用规范的表达方法表达出来（例如 E-R 图）。

如何做逻辑数据库设计

逻辑数据库设计最后要提交 逻辑数据库设计报告。应当包含数据库的逻辑设计。

E-R 图向关系模式的转换

  • E-R 图的实体转换为关系
  • E-R 图的属性转换为属性
  • E-R 图的关键字转换为主键
  • E-R 图的复合属性应当分解成多个属性或者合并车一个关系
  • E-R 图的多值属性应当和主键一起 形成一个新的关系
  • 一对一联系，且双方都是部分参与时，可以将联系定义为新的关系，属性为双方的主键
  • 一对一联系，且一方是全部参与时，可以将另一方（可能是全部参与或者部分参与）作为全部参与的这一方的属性
  • 多对一联系，将单方参与实体的主键作为多方参与实体的属性
  • 多对多联系，应当新建一个关系，其属性为双方的主键
  • 弱实体的主键应当包含依赖的强实体的主键
  • 分类实体对应的关系应包含一般实体的主键
  • 如果分类是完全的，则可以不建立一般实体对应的关系
  • 对于多元联系（一个联系关联多个实体），可以
    • 引入新的关系表示这个联系，关系应继承各个实体的主键，将其作为主键
    • 引入新的关系，并使用自己独立的 ID 作为主键，然后引入关联的各个实体的主键作为普通的属性
    • 转换为多个二元联系
